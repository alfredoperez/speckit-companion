Spec-Context Tracking & Viewer Status Feedback

spec.md · the what and the why

Feature Branch060-spec-context-tracking
CreatedApril 13, 2026
StatusDraft
InputImprove how SpecKit Companion records spec lifecycle state in .spec-context.json and how the viewer reflects that state, so users get accurate, consistent, real-time feedback about what is happening in a spec.

Problem Context

Today .spec-context.json is inconsistently populated across workflows. The viewer derives step status partly from file existence, which produces false positives, like a plan.md existing from a template being treated as "planned." The stepper, header badge, footer buttons, and pulse indicators drift out of sync with reality. Users cannot trust what the viewer tells them.

054archive-button-left · Companion-driven: reached completed after only specify + plan, with noisy back-and-forth transitions.
055fix-bullet-rendering · Terminal-only: context is just { "status": "completed" } with no lifecycle data.
056fix-list-spacing · SDD Fast: created all files in one run, but context only records specify started.
058floating-toast · SDD: status: completed while tasks.completedAt is null (contradictory).

Summary

Make .spec-context.json the single source of truth for spec lifecycle state, with a canonical schema. All four workflows write the same shape via standardized prompt blocks. The viewer derives badges, pulse, highlight, and footer button visibility solely from this context, never from file existence.

Research

key decisions
Decision · single status field
Chosen: one spec-wide status over per-step statuses.
Why: per-step status is the root of the "viewer feels wrong" reports.
Alternatives: per-tab status (rejected — drift), derived-only (rejected — can't represent draft).
Decision · append-only transitions
Chosen: never mutate prior transition entries.
Why: preserves an auditable lifecycle even across regenerates.
Alternatives: last-write-wins (rejected — loses history).
Decision · back-fill minimal context
Chosen: on first open of a terminal-only spec, write only what's verifiable.
Why: terminal runs can't be intercepted; never infer completion.
Alternatives: infer from files (rejected — the original bug).

Schema

contracts/
spec-context.schema.json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["workflow", "specName", "status", "history"],
  "properties": {
    "status": { "enum": ["draft", "specifying", "specified",
      "planning", "planned", "tasking", "ready-to-implement",
      "implementing", "completed", "archived"] },
    "history": { "type": "array", "items": { "$ref": "#/$defs/event" } }
  }
}
