02 · Rationale

Why this reads better

The proposal adds hierarchy only where the document has repeatable meaning. Ordinary prose remains ordinary prose.

01

Scan before reading

IDs, confidence, coverage, and groups establish a fast information scent. A reader can find the risky requirement without parsing every paragraph.

02

Structure carries meaning

WHEN and THEN, observed and inferred, covered and uncovered are semantic distinctions—not styling decoration.

03

Honesty stays prominent

The “Uncovered” section starts with a plain-language scope summary, then lets readers inspect the exact files and reasons on demand.

Implementation-ready ticket draft

Scoped as a capability-reader enhancement. The authoring format remains Markdown, existing feature-spec rendering remains unchanged, and every component must preserve line-level commenting.

Capability Reader — semantic document components

Problem. Living specs currently render recurring structures as generic headings, paragraphs, and bullets. The content is technically present but hard to scan, compare, or qualify—especially requirements, acceptance scenarios, and the “Uncovered” evidence boundary.

Outcome. In living-spec mode, recognized structures render through a small set of accessible reader components. Unknown Markdown keeps the existing renderer as a lossless fallback.

Acceptance criteria

  1. Living-spec purpose text renders as a distinct lead component; the same text is never duplicated elsewhere in the body.
  2. Requirements render in grouped rows with identifier, plain-language title, body, observed/inferred confidence, clarification state, and coverage state when those facts exist.
  3. Acceptance scenarios visually separate conditions from outcomes while preserving their original order and exact wording.
  4. The Uncovered section begins with a count and plain-language scope statement, then groups file-level omissions by reason in keyboard-accessible disclosures.
  5. No component invents metadata. Missing confidence, coverage, source, or counts are omitted—not shown as zero or guessed.
  6. Every semantic block remains addressable by source line and supports the existing add, restore, edit, and delete comment flows.
  7. The reader uses the viewer’s owned dark/light/high-contrast tokens, honors reduced motion, and passes keyboard and contrast checks.
  8. Feature specs and unrecognized Markdown remain behaviorally unchanged; rendering failures fall back to the current Markdown output instead of blanking the document.
  9. Storybook stories cover complete, sparse, inferred, clarified, uncovered, narrow, light, dark, and high-contrast states.

Component map

  • CapabilityDocumentLiving-mode composition and safe fallback boundary.
  • PurposeCalloutOne-sentence reader orientation.
  • RequirementGroupIntent grouping and group count.
  • RequirementRowID, title, body, confidence, coverage, source.
  • ScenarioCardOrdered conditions and outcomes.
  • EvidenceSummaryCoverage boundary and review scope.
  • EvidenceDisclosureFiles grouped by omission reason.

In scope

  • Typed living-spec block model
  • Preact reader components
  • Source-line/comment adapters
  • Stories and renderer tests

Out of scope

  • Changing Markdown file format
  • Editing capability content in cards
  • Recomputing drift or coverage
  • Restyling feature specs globally
