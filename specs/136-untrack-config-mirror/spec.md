# Untrack the Companion config mirror

## Overview

The Companion's per-developer template-profile choice is persisted to `.specify/companion.yml`, but that file is checked into version control while the extension rewrites it on every activation and setting change. This makes the file permanently show as modified for anyone whose setting differs from the committed value, and leaks one developer's local choice into everyone else's checkout. This feature makes the config mirror machine-local: untracked, gitignored, and created on demand by the extension.

## Functional Requirements

- **FR-001**: The repository MUST stop tracking `.specify/companion.yml`; the file MUST be removed from version control without deleting any developer's existing working copy.
- **FR-002**: `.specify/companion.yml` MUST be listed in `.gitignore`, accompanied by a comment identifying it as a machine-local mirror of the template-profile setting written by the extension.
- **FR-003**: Changing the template-profile setting or reloading the editor MUST NOT cause any tracked file to appear as a modified, unstaged change as a result of the mirror rewrite.
- **FR-004**: The extension MUST create `.specify/companion.yml` on demand when it is absent, so a fresh clone that lacks the file works without error.
- **FR-005**: Switching between the standard and turbo template profiles MUST continue to work exactly as before, including the project-default selection and per-spec profile pinning.
- **FR-006**: A developer's template-profile choice MUST remain local to their own checkout and MUST NOT propagate to other checkouts through version control.
- **FR-007**: Every piece of documentation that references the mirror file MUST describe it as machine-local and gitignored, including the README "Template Profiles" section, `docs/template-profiles.md`, `docs/capture-and-timing.md`, and `speckit-extension/docs/install.md`.

## Success Criteria

- **SC-001**: After changing the template-profile setting and after reloading the editor, `git status` reports zero modified tracked files attributable to the mirror.
- **SC-002**: A clean checkout with `.specify/companion.yml` absent activates the extension and produces the file with no error surfaced to the user.
- **SC-003**: Standard and turbo profiles can each be selected as the project default and pinned per-spec, with the same resulting behavior as before this change (no regression in profile routing).
- **SC-004**: One developer changing their profile setting produces no diff in any other developer's checkout.
- **SC-005**: 100% of the documentation locations that mention the mirror file describe it as machine-local/gitignored after the change.

## Assumptions

- The mirror file is read **only** by the VS Code extension; no spec-kit script, command body, or preset depends on it, so untracking it cannot break the spec-kit side (consistent with the same treatment `.specify/presets/` received previously).
- The file is regenerated by the extension on activation, so its absence on a fresh clone is non-fatal — untracking is safe precisely because nothing breaks when it is missing.
- The file stays at its current location `.specify/companion.yml`; it is not relocated under `.specify/extensions/companion/` (that directory is gitignored and regenerated on install, which would wipe persisted config on a forced reinstall).
- The extension continues to persist the setting via the mirror file rather than reading the VS Code setting directly everywhere, to avoid coupling the VS Code-free dispatch path to the VS Code API.
- The standard/turbo profile names reflect the current naming after the lean→turbo rename; this change coordinates with but does not depend on that rename.
