# Plan — Untrack the Companion config mirror

## Summary

Stop tracking `.specify/companion.yml` in git so a developer's local template-profile choice no longer leaks across checkouts and no longer shows as a permanent unstaged modification. The fix is a version-control + documentation change: `git rm --cached` the file, add a gitignore entry with an explanatory comment, and update the four docs that mention the file to describe it as machine-local/gitignored. No extension source changes are needed — `writeTemplateProfile()` already creates the file on demand (`mkdirSync` + write) and `readTemplateProfile()` already returns `undefined` when it is absent, so a fresh clone without the file works today.

## Technical Context

- **Language/version**: TypeScript 5.3+ (ES2022, strict) — but no `.ts` changes expected.
- **Primary artifacts touched**: `.gitignore`, git index (`.specify/companion.yml`), Markdown docs.
- **Storage**: `.specify/companion.yml` (per-workspace YAML mirror of `speckit.companion.templateProfile`).
- **Testing**: existing reconciler tests (`companionPresetReconciler.test.ts`) already cover absent-file read and on-demand write; no new tests required. Verification is via `git status` after a setting change + reload.
- **Target platform**: VS Code extension host (Node.js) + spec-kit CLI workspace.
- **Constraints**: must not delete any developer's existing working copy of the file (`--cached` only); file stays at `.specify/companion.yml` (not relocated under the gitignored `.specify/extensions/companion/`).

## Approach & Structure

Organized by artifact, in order of attack:

1. **`.specify/companion.yml`** (git index) — `git rm --cached .specify/companion.yml`. Removes it from version control while leaving the working copy on disk untouched. This is the change that satisfies FR-001 and is the parent of FR-003/FR-006.

2. **`.gitignore`** (around the existing `.specify/presets/` / `.specify/extensions/companion/` block, ~line 44–47) — add `.specify/companion.yml` with a one-line comment identifying it as a machine-local mirror of the `speckit.companion.templateProfile` setting, written and regenerated by the extension. Mirrors the wording style already used for the adjacent generated/local-only entries. Satisfies FR-002.

3. **No source changes** — confirm (do not edit) that the existing write/read path already handles absence:
   - `src/features/settings/companionPresetReconciler.ts` — `writeTemplateProfile()` does `fs.mkdirSync(dir, {recursive:true})` then writes; `readTemplateProfile()` returns `undefined` when the file is missing. Satisfies FR-004 with zero new code.
   - `src/extension.ts` (~line 251) — activation already mirrors the project default into the file, regenerating it on a fresh clone.
   - `src/features/specs/profileDispatch.ts` — only reads the mirrored value; unaffected. FR-005 (standard/turbo routing + per-spec pinning) is preserved because nothing in the routing path changes.

4. **Documentation** (FR-007) — update each location that names the file to call it machine-local/gitignored:
   - `README.md` — "Template Profiles" section.
   - `docs/template-profiles.md` — the profile/selection-model reference.
   - `docs/capture-and-timing.md` — wherever the mirror is described as an install/config path.
   - `speckit-extension/docs/install.md` — install-path mention.

   These are the four files the spec enumerates; a `git grep "companion.yml"` over `README.md docs/ speckit-extension/docs/` confirms the set (the `src/` matches are code comments, not user docs, and stay as-is).

## Out of Scope

- Relocating the mirror file to a different path.
- Changing how the profile setting is stored, read, or routed (the VS Code-free dispatch path keeps reading the YAML mirror, not the VS Code setting directly).
- Any change to the standard↔turbo profile names or the lean→turbo rename (this change only coordinates with it).
- Editing the `src/` code comments that mention `companion.yml` — they remain accurate.

## Decisions

- **Untrack, don't relocate.** Keeping the file at `.specify/companion.yml` (vs. moving it under `.specify/extensions/companion/`) avoids wiping persisted config on a forced `--dev` reinstall, which regenerates that directory.
- **No code change for on-demand creation.** The existing reconciler already creates the file when absent, so untracking is safe and self-healing; adding bootstrap code would be redundant.
- **Same treatment as `.specify/presets/`.** This follows the precedent set when the locally-installed preset copies were gitignored — a machine-local, regenerated artifact that shouldn't be tracked.
