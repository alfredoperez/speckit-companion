Pipeline + the sdd-lean Preset

Execution & test plan — bringing the SDD-lean document shape into spec-kit, ready for review before commit/PR.

The shape, without the ceremony. A native spec-kit preset reshapes the stock pipeline to the SDD-lean shape — specs with no user-story section, lean plans, and tasks on a files/dependencies axis — and four opt-in /speckit.companion.* commands give the same shape on demand. 23 of 26 tasks done; every automated check is green; the 3 that remain are live spec-kit runs only a human can drive.
How it works · two levers
Lever 1 · preset

sdd-lean replaces: the four core commands. Once installed, stock /speckit.specify|plan|tasks|implement emit the lean shape. On by default.

Lever 2 · opt-in

/speckit.companion.specify|plan|tasks|implement emit the same shape regardless of preset state. Shared bodies, kept identical by a drift guard.

Control · toggle

speckit.features.sddLean (default on) reconciles the preset; persisted to .specify/sdd.config.yml as the source of truth.

Stock vs SDD-lean
Stock spec-kit spec.md
  • User Stories (P1/P2/P3) with priorities
  • Acceptance-scenario tables per story
  • Tasks grouped under each user story
  • Multi-phase plan scaffolding
SDD-lean spec.md
  • Overview · Functional Requirements
  • Success Criteria · Assumptions — no stories
  • Tasks on a files/dependencies axis
  • Lean plan: summary, context, approach
Decisions baked in · research corrections
1

Use the native preset engine — don't build one

spec-kit already ships a preset subsystem with a resolution stack (project → presets → extensions → core), priority ordering, and composition. We ride it.

2

specify preset add, not preset use

The backlog's command string didn't exist. Real verbs: add / enable / disable / set-priority / remove / resolve.

3

Reshape via command overrides, not raw templates

Core commands embed their own structure, so only a replaces:-command override actually changes what stock /speckit.specify emits. (The shipped lean preset proves the pattern — but it keeps user stories, so sdd-lean is distinct.)

4

Off = remove, never just disable

Command overrides register into agent dirs at install time; disable only skips runtime file resolution. Only remove restores the stock commands — so the toggle's off-path maps to remove.

5

Dual-extension split, isolation preserved

Preset + commands live under speckit-extension/ (own README/CHANGELOG/version). Only the VS Code setting touches root package.json + src/. The .vsix ships nothing from .specify/.

Test runbook · the 3 remaining manual E2E

Prerequisite

A GitHub-source spec-kit CLI (specify preset --help works) in a project initialized with specify init. Full steps live in specs/132-sdd-lean-pipeline/quickstart.md.

  1. Install (T-setup). specify extension add ./speckit-extension --dev, then specify preset add --dev ./speckit-extension/presets/sdd-lean. Expect specify preset list to show sdd-lean enabled and specify preset resolve speckit.specify → sdd-lean.
  2. Preset path · US1 / SC-001 (T008). Run stock /speckit.specify "Add a thing" → spec.md has no user-story section. Spot-check /speckit.plan (lean) and /speckit.tasks (files/deps axis).
  3. Opt-in path · US2 / SC-003 (T014). specify preset disable sdd-lean, then /speckit.companion.specify "Add a thing" → same lean shape; its section headings match the preset-path output.
  4. Toggle off · FR-007 / SC-004. specify preset remove sdd-lean (or set speckit.features.sddLean false in VS Code) → stock /speckit.specify returns the stock template with a user-story section, no residual lean sections.
  5. Composition · SC-005 (T025). Add a 2nd preset that also replaces speckit.specify, specify preset set-priority sdd-lean 5, then specify preset resolve speckit.specify → sdd-lean wins deterministically across repeated runs.
