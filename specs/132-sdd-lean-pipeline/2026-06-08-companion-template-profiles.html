Companion Template Profiles

Two profiles for the spec-kit pipeline. Standard = the stock commands, unchanged, with better timing. Lean = the same commands with specific sections trimmed or replaced. Here's exactly what changes, command by command, section by section.

Overview

SpecKit Companion runs the spec-kit pipeline in two profiles (plus off). Pick a default for the project, and flip a single spec to lean from its right-click menu.

standard — default
  • Stock commands — same sections, same files
  • Nothing reshaped; only better timing added
  • Closest to upstream spec-kit
lean — opt in
  • Same commands, trimmed — no user stories, files/deps tasks
  • Smaller spec folder — spec/plan/tasks + requirements checklist; side files on demand
  • Modifies the command bodies; document templates left stock

off = plain stock spec-kit, no overrides at all.

Choosing a profile · two levels

1 · Project default

One setting picks the default for the whole project. Mutually exclusive — choosing one removes the other preset.

2 · Per-spec override

A spec's right-click menu → Template Profile → Standard / Lean overrides the default for that one spec, records profile in its .spec-context.json, and the GUI dispatches the matching commands through plan → tasks → implement (across every dispatch path — footer, palette, sidebar).

# 1 — project default (the preset; reshapes stock commands) speckit.companion.templateProfile = "standard" | "lean" | "off" → .specify/companion.yml # 2 — per-spec override, set from the spec's right-click menu .spec-context.json → "profile": "lean" → this spec's plan/tasks/implement follow it

Mechanism: the project default rides the spec-kit preset (reconciled on activation and on change); the per-spec lean path rides the namespaced /speckit.companion.* commands, resolved by one shared helper across the footer / palette / sidebar so every path honors it. off uses neither.

Why the shape lives in commands, not templates

Lean modifies the command bodies, not the document templates — on purpose. spec-kit only resolves a preset's template override when a setup script runs the resolver, so a lean spec-template.md would silently do nothing for specify (the agent copies the core template directly), while working for plan/tasks. Command overrides apply to every command uniformly, so the shape goes there. Trade-off: in lean mode the on-disk spec-template.md still looks stock — but lean never reads it. Captured in docs/template-profiles.md.

Better timing · both profiles

Both profiles bake the same timing rules into every command body, so durations stay honest for any dispatcher (not just the GUI). Fixes three logged bugs:

Self-close

Each step writes its own complete when its work ends — specify no longer borrows plan's start as its end.

No duplicate start

A repeated same-step start is deduped at write time instead of doubling the journal.

Live cadence

One fresh date -u per substep/task + a per-task complete — no end-of-run burst with 0ms gaps.
