# Implementation Plan: Auto-load living specs into specify & plan (LS·2)

**Spec**: [spec.md](./spec.md) · **Issue**: #362 · **Builds on**: #361 (LS·1)
**Surface**: spec-kit extension (`speckit-extension/`) — versions/docs are the extension's own, NOT root.

## Approach

LS·1 shipped the resolver (`speckit-extension/scripts/resolve-spec-paths.py`) and the typed `livingSpecs` reader on `companion_config.py`. LS·2 wires that resolver into the **specify** and **plan** command bodies (via node assembly) so the assistant auto-loads the matching capability living specs and records which it loaded, and the plan step reuses that record.

Three moving parts:

1. **Recording write path** — `write-context.py` gains a dedicated, non-lifecycle flag to record the loaded capability names as a list (since `--set` coerces to scalars only). Stored under a top-level `livingSpecs` object: `{ "loaded": [<names…>] }`. The strict schema already has `additionalProperties: true`, so nothing in `spec-context.schema.json` needs to change for it to validate; `check_capture.py` only asserts lifecycle fields, so it stays green. The merge reads the existing record, de-dups, and never touches lifecycle keys.

2. **Node bodies** — a new `load-living-specs` node under `nodes/specify/` and an edited `nodes/plan/gather-context.md` whose instruction prose tells the assistant to: detect `livingSpecs.enabled`, run the resolver in `--changed … --json` mode over the in-scope files, read each matched `capabilities/<name>/spec.md` most-specific-first, and record the loaded names via the new write-context flag. Plan's variant prefers the recorded list over re-resolving. All steps are framed best-effort: missing config/spec/resolver → skip, never block; never write a living spec.

3. **Assembly + golden** — re-run `assemble-nodes.py` to regenerate `commands/speckit.companion.specify.md` and `…plan.md`, then re-bless the golden with `capture-golden.py`. specify & plan are already in `check-shape-parity.py`'s `INTENTIONALLY_CHANGED` set, so golden equality is exempt for them, but re-blessing keeps the snapshot honest (per #319 lesson).

### Recording shape (the key design call)

`.spec-context.json` gains:
```json
"livingSpecs": { "loaded": ["checkout-cart", "checkout"] }
```
Most-specific-first order preserved from the resolver. The new write-context flag merges this in without rebuilding the record (#256 lesson) and refuses if asked to touch a lifecycle key. Plan reads `ctx["livingSpecs"]["loaded"]` first; only if absent/empty does it re-resolve.

### Node-assembly guardrails (#319)

- The new specify node injects its work as its own numbered top-level step only if it slots cleanly into the existing 1..N sequence; otherwise it uses an unnumbered "**Load living specs (best-effort).**" note so the assembled body does not double-number. Verify by reading the ASSEMBLED `commands/speckit.companion.specify.md`, not just the node.
- After the edit: `assemble-nodes.py` (write), then `assemble-nodes.py --check`, `capture-golden.py`, `check-shape-parity.py` all green.

### Opt-in / safety

Every load action is gated on `livingSpecs.enabled` being true AND the resolver returning matches AND the spec file existing. Any miss → no load, no recording, stock behavior. Never writes a living spec from specify/plan.

## Constitution Check

No constitution file gates this; the change respects extension isolation (only `speckit-extension/**`, bench, specs, vault — never root version/README, never `.claude/**` runtime deps).

## Risks

- **Schema drift** — mitigated: store under additive `livingSpecs` key; schema already permits it; pytest asserts the capture eval's lifecycle fields are untouched.
- **Golden parity trip** — mitigated: specify/plan already exempt; re-bless anyway.
- **Live-AI sandbox infeasible from this context** — fall back to a deterministic LS2 demo (resolver + recording write path) with honest `mode: deterministic` and `verdict: INCONCLUSIVE` for any un-runnable live step.
