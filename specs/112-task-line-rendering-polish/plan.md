# Implementation Plan: Task-Line Rendering Polish

**Branch**: `112-task-line-rendering-polish` | **Date**: 2026-05-27 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `/specs/112-task-line-rendering-polish/spec.md`

## Summary

Fix two spec-viewer rendering regressions in the task-line layer:

1. **Issue A — Trailing break after wrapping tasks**: An empty `.line-comment-slot` div with `flex-basis: 100%` always occupies a full flex row in the `li.task-item` flex container, even when it holds no comment. On single-line tasks the row collapses invisibly; on wrapping tasks the slot's empty row creates a visually distinct trailing gap. Fix: collapse empty slots via CSS (`:empty` selector or `display: none` when height is 0).

2. **Issue B — Hover content shift**: The `:hover` rule on `li.task-item` applies `padding: var(--space-1) var(--space-2)` which is a four-side shorthand, replacing the idle state's `padding-right: 24px` (reserved for the absolutely-positioned `+` button) with the much smaller `var(--space-2)` (8 px). The resulting right-padding reduction narrows the text column, reflowing wrapping text. Fix: explicitly carry `padding-right: 24px` into the hover rule.

3. **Storybook coverage**: Add task-row stories (idle/hover × single-line/wrapping + paragraph baseline) so future regressions surface in visual review.
