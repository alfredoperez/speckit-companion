One ladder, five rungs, same shipped code

A 15-cell benchmark built the same feature at three sizes through five SpecKit Companion workflow configurations — an "adoption ladder" where each rung changes exactly one variable. Every cell ran through the real VS Code extension, then was graded by a deterministic harness plus an independent judge agent that didn't write the code.

2026-06-10·SpecKit Companion· 15/15 shipped 5 modes × 3 sizes run-in-folders · /bench-prep · /bench-capture
Thesis: all five rungs ship correct, convention-following, fully-tested code — every time, at every size. Correctness never moves. What moves is process ceremony and progress visibility. The lean rungs (turbo, fast-path) cut spec ceremony roughly in half and write zero throwaway side files; fast-path additionally collapses the whole pipeline on simple work and gracefully steps back up to the full pipeline when the work is genuinely complex.

Result 3 — the headline: fast-path is adaptive

The most interesting finding isn't that fast-path is faster — it's that it knows when to be. It folds the plan/tasks pipeline into specify only when the change is genuinely simple, and steps back up to the full pipeline when it isn't.

easy + medium → it folds
  • plan.md and tasks.md = 0 lines, 0 tasks — collapsed into specify.
  • Finishes in ~6.5 min — less than half every other rung.
  • Tightest diff, zero out-of-scope, still 5.0 rubric.
hard → it declines to fold
  • Classifies the routed, multi-surface Tags feature as non-simple.
  • Runs the full plan→tasks pipeline — 12 tasks, 14m12s.
  • Identical to turbo (14m48s, 12 tasks). No penalty for the safety check.

Result 4 — progress visibility (the thing speckit can't buy)
speckit — blind
  • Stock upstream writes no usable lifecycle history.
  • The GUI has no step, no status, no resume to render.
  • You find out where a spec stands by reading files.
companion (4 rungs) — tracked
  • Full canonical history[]: a finish per step and per task.
  • Capture eval 14–17 ✓ / 0 ✗ on every companion cell.
  • Honest live timing — the GUI shows real progress and can resume.
The pick
speckit
  • Identical shipped code…
  • …but heaviest doc ceremony (4 side files, longest specs).
  • Zero progress capture — blind to the GUI.
turbo / fast-path — sweet spot
  • Same correctness, same feature, full capture.
  • ~half the spec ceremony and diff, no throwaway side files.
  • fast-path = turbo + a free speedup on simple work, no penalty on hard.

companion-logs is the entry rung — turn on visibility, change nothing else. companion-standard is the middle road: capture on, but you keep the verbose upstream artifact shape.

Caveats — what this run hardened, honestly

Dogfooding the bench for real surfaced (and fixed) measurement issues that matter more than any single number:

  1. Artifact ceremony was undercounted. The harness only counted spec/plan/tasks lines — never the lines inside research.md / data-model.md / quickstart.md / contracts/ / checklists/. Fixed: a recursive artifact-total-lines metric now counts every file the spec produced. This run predates it, so the side-file numbers here are by presence (4 vs 0); the next run quantifies the true line gap — which favors the lean rungs even more.
  2. A medium blast-radius false-positive. The idiomatic new lib/dueDate.ts helper (judges praised it) was wrongly flagged "out-of-scope." Fixed: medium scope now allows new lib/ files, the way hard already allows new components/.
  3. speckit isn't perfectly blind. The always-on VS Code extension writes some timing even for stock upstream — so speckit shows partial timings, and its medium run is missing the implement timestamp, making that one total non-comparable.
  4. Implement step never auto-closes (#244). The VS Code manual flow leaves implement at implementing instead of advancing to implemented. Mode-agnostic — affects every rung equally, filed separately. Not a fast-path bug.

Repeatable, and it accumulates

The round runs from the five sandbox folders directly — /bench-prep <size> arms and opens them, you build in the real extension, /bench-capture <size> judges + measures + banks a row into bench/stats.jsonl and regenerates REPORT.md. As fast-path, capture, and future features land, re-running shows whether the lean rungs keep holding the line on correctness — or start trading it away.
