Parallel pipeline, sharper output, and a readability call

What we shipped this pass, and what to do about the spec/plan/tasks output looking like “lists of code.” June 16.

1What we shipped

The pipeline now does independent work at the same time instead of one step after another, and it checks its own plan for bugs before building. All validated by running the real steps by hand and watching what actually happened. It lives on a branch, uncommitted.

Work runs in parallel waves

The task list is now grouped into waves of genuinely independent pieces, and the build does a whole wave at once (one helper per piece) before moving on. On the hard test feature a five-piece wave finished in about the time of its slowest piece, not the sum.

Planning investigates in parallel too

Before designing, the plan step now reads different parts of the codebase side by side (one helper per area) and gets back short summaries instead of reading the whole app into one context. That banks time and context budget.

A built-in skeptic reviews the plan

When the task list is generated, a small panel of reviewers attacks the spec and tasks for the bugs lean specs ship. It found a real one (deleting a todo left its tags orphaned), added the fix to the task list, and stayed quiet on a clean spec. The banked time above is what pays for this, so quality rises without a net slowdown.

Side-by-side work stays coherent

When two helpers write files that share an interface, they used to mismatch at the seam. Now the task list pins the shared shape once and hands it to each helper, and the build reconciles after each wave. The pair that broke twice before now lands clean on the first try.

2The output readability question

You flagged that Companion output is “all lists” and dense with inline code, and asked whether to move closer to spec-kit. Three agents examined the real spec, plan, and tasks files against the stock templates and the viewer code. The headline finding is not what I expected.

The surprise — the viewer is amplifying the problem

The “code-span soup” in your screenshot is not only the markdown. The viewer promotes any path-shaped span like src/store/tags.tsx into a bordered, icon-prefixed, accent-colored pill. Technical Context and Approach are exactly where file paths cluster, so ten bullets each open with a glowing pill plus several more code spans. That is the soup. You control that rendering, so it is the cheapest, highest-impact fix, and it also repairs every plan already on disk.

What the markdown is doing

Technical Context today

State: reducer + context per slice in src/store/, exposed via a useXxx() hook. New TagsProvider/useTags() mirrors TodosProvider/useTodos().

The issue

Six code spans in one bullet, and only src/store/ is a real path. The rest are ordinary nouns the model chose to backtick. The node never tells it not to, so it over-codes by default.

Two real problems, separate from the pills: every metadata section is a Label: value bullet list (that wants to be a small table), and the requirement lists are tabular data, an ID plus a sentence, rendered as fifteen identically-shaped bold bullets so the eye has no anchor.

Should we move closer to spec-kit?

Short answer: no, not in shape. Stock spec-kit is longer and more templated, not more readable. Its per-story “Why this priority / Independent Test / Acceptance Scenarios” scaffold is exactly the bloat Companion was right to cut, and Companion’s prose Overview is better than anything stock opens with. Keep the lean shape and keep dropping user stories. Borrow only three cheap things stock has and Companion dropped: one plain sentence of narrative per major flow, a short named-Entities list, and a few Edge Case questions.

What to actually change

Two levers. The viewer fixes what is already on disk and is fully yours to change. The authoring prompts fix new docs at the source. Lead with the viewer.

My recommendation

Keep Companion’s lean, prose-led shape, do not regress toward spec-kit’s templates. The output reads as “lists of code” mostly because the viewer you own is dressing plain markdown up as code, and secondarily because the plan node over-backticks. Fix the viewer first: suppress the file pills in metadata sections and turn label/value and requirement lists into tables. That alone repairs every existing doc and the screenshot you sent. Then add backtick discipline and one-sentence intros to the nodes so new docs stop creating the problem. The three small spec-kit borrowings are a nice-to-have, not the fix.
