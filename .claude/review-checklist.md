# Review checklist — scan every diff against this

The review subagent in `/ship-ticket` and `/fix-tickets` reads this **before** reviewing, and checks the diff against it. These are recurring bug classes earlier reviews caught — phrased as *scans*, not narrative. Keep it that way:

- A line earns its place only if it's **checkable** (a reviewer can verify it against a diff), **recurring or high-cost**, and **phrased as a check** ("flag X when Y"). Edit an existing line before adding a near-duplicate.
- Universal authoring rules live in **`CLAUDE.md`** (Webview & rendering invariants, Code Comments, Design tokens) — scan the diff against those too; they're not repeated here.
- One-off insights, style preferences, and loop mechanics do **not** belong here (loop mechanics live in the command files).

---

## First pass — always

- **CLAUDE.md conventions.** Scan for any violation of the *Webview & rendering invariants*, *Code Comments*, *Design-token*, and *Markdown formatting* rules in `CLAUDE.md` — the last one covers **every** `.md` in the repo (a hard-wrapped `docs/*.md` plan shipped in #435 because this scan only named the first three). (#435)
- **Comments.** Flag added comments — **including Python docstrings and test docstrings** — that restate the code, narrate a fix's history ("""The #432 regression itself…"""), carry a spec/PR/finding id (`// per #182`), or run past one line. Default is no comment. (#432)
- **Diagnostic logs.** Flag `console.log`/bracket-tag debug lines left in; structural `logError`/catch-block logging stays.

## Capture / `.spec-context.json`

- **Terminal status is `implemented`, never `completed`.** `completed` is the user's Mark-Completed action only; the capture script refuses to write it. Flag any code/doc that forces `completed` on autonomous finish. (#208, #244)
- **Fast-path lands at `tasks` + `ready-to-implement`, not "at implement."** Flag docs/changelogs/command bodies that say a fast-tracked spec "lands at implement." (#237)
- **Upsert by MERGING onto the existing record, never rebuilding it.** A writer that does `record[id] = {…}` each call drops reader-supported fields (`concerns`) and erases prior `did`/`files` on a bare re-journal. Read existing → copy → set only new non-empty fields. (#256)
- **Derive UI fields from the FULL discriminator, not field-presence.** Gate a surfaced optional field (e.g. active task id on a row) on step + finish/kind, not just `field != null` — keep a legacy `kind == null` fallback. (#238)
- **A generic field-setter/merge onto `.spec-context.json` must refuse lifecycle keys** (`history`/`status`/`currentStep`/`transitions`). A bare `ctx[key] = value` loop (e.g. `write-context.py --set`) otherwise lets `--set status=completed` bypass the mark-complete safeguard and corrupt the log. Deny-list lifecycle keys with a stderr note; cover with a protected-key test. (#309)
- **A `- [ ]`/`- [x]` scan must anchor to line start AND skip fenced blocks and inline code spans.** An unanchored regex counted the legend line inside `tasks.md`'s own code fence as a real task — 7 of 210 specs permanently mis-reported (one stuck at 60% with every task done), and one fence-blind counter held back phase-completion notifications forever. There is exactly one counter: `src/core/utils/taskCheckboxes.ts`. Flag any diff that writes a fresh checkbox regex instead of calling it. (#443)
- **Task completeness compares per-occurrence marker counts, never collapsed sets.** `set(all) <= set(done)` reads 100%-done when a duplicate id has one marker checked and one not; use `len(done) == len(all)` on the per-occurrence lists. A completion/step-close gate also needs BOTH signals — tasks.md at 100% AND every task journaled — not either alone, or a journaled-but-unchecked task closes the step while status stays `implementing`. (#317)
- **GC a derived/append file (e.g. `.spec-context.events.jsonl`) only after its data is durably folded AND only at a transition that blocks re-creation.** Two failure modes: unlinking before the `atomic_write` that persists the fold (a crash drops both), and unlinking at a step-close that a later parallel-wave append can re-create (the straggler line is then orphaned). Remove it at the terminal `completed` write — after `CROSS_STEP_TERMINAL` starts refusing appends — and fold any pending lines (idempotent) before the unlink. (#349)
- **A new `status`-writing path must guard forward-only with `_is_more_advanced`, not just the terminal (`completed`/`archived`) guard.** A writer that sets `status` to a step's canonical value after only the terminal check drags a spec backward when an earlier step is re-advanced (a re-run, a double-fired hook) — e.g. `--advance plan` on a `ready-to-implement` spec regressed it to `planned`. Flip `status`/`currentStep` only when the spec hasn't already moved past the step; the idempotent history append is safe regardless. (#351)

## Node assembly (`speckit-extension/nodes/`)

- **A node that injects a step into a command that keeps numbering downstream must use a sub-bullet or unnumbered note, not a new top-level `N.`** — the assembled body concatenates node bodies, so a fresh `2.` in one node lands next to the next node's `2.` and double-numbers. Check the ASSEMBLED command, not just the node. After any node/part edit, re-bless golden (`capture-golden.py`) and confirm `assemble-nodes.py --check` + `check-shape-parity.py` pass. (#319)
- **Prose telling a capable provider to run work concurrently must name who serializes the `.spec-context.json` write** — "journal each as it finishes" under per-task subagents reads as concurrent writes (the race the timing part warns about); say the MAIN agent records one at a time, foreground. (#319)

## Command-family & extension gating

- **Guard a command/namespace family by shared PREFIX, not an enumerated subset.** An enumerated map of the 4 pipeline commands let a 5th (`speckit.companion.mark-complete`) slip the guard. Key off the `speckit.companion.` prefix; suppress members with no stock twin rather than dispatching them. (#300)
- **Gate `/speckit.companion.*`-dependent UI on the extension dir** (`.specify/extensions/companion/`), not on preset presence — a preset-only project must read as "not installed." Check `isCompanionInstalled()` directly at *every* call-site a namespaced command could dispatch. (#218, #234)
- **Before making an activation-time step unconditional, confirm its inputs/paths always exist.** Removing a gate made a preset-add run on every activation in projects lacking the companion dir → failing command + error log each activation. Gate on the on-disk install signal and rerun from the existing file watcher. (#300)
- **Close the implement step from the always-on `tasks.md` watcher, not terminal/hook paths** — implement has no "next step," IDE-chat returns no terminal handle, stock mode has no hook. (#244)

## Keys, collisions & identifiers

- **Classify a command by its VERB (first token), not a substring of the whole command+args.** `/companion/i.test(command)` matched a stock command whose arg path contained "companion" (`/speckit.plan specs/123-companion-feature`) and mis-slimmed its preamble, dropping the only capture source. Split the verb off first (`command.trim().split(/\s+/, 1)[0]`) and test that. (#352)
- **A synthetic/follow-on map entry needs a DISTINCT key.** Reusing `tempFileSet.id` for a staged-images entry clobbered the temp-set entry and leaked its dir; use a derived key (`<id>-staged-images`). (#208)
- **Guard synthetic list entries against name collision.** Skip + warn when injecting a reserved-named entry (workflow option, etc.) if a user entry already uses that name — else duplicate DOM `<option>`s + last-write-wins `Map` clobber. (#218)
- **Persist only resolvable identifiers.** Don't write a UI-only synthetic name (`speckit-turbo`) into `.spec-context.json`; persist the resolvable base (`speckit`) + a separate pin (`profile: turbo`). (#218)

## Types & data boundaries

- **Enum-by-TYPE is not enum-by-DATA — coerce at the privacy/telemetry boundary.** A field typed `'a'|'b'` sourced from user-authored data (`settings.json`, `.spec-context.json`, custom-workflow step names) can carry arbitrary text. Allow-list at the emit boundary; don't trust the declared type. **And never key a LOOKUP on such a value with a plain object literal** — `{ specifying: 'specify', … }[status]` resolves `constructor`/`toString`/`__proto__` off `Object.prototype` and returns a truthy function, so an arbitrary `status` in `.spec-context.json` reads as a legitimate hit (this silently suppressed the viewer's forward button). Use a `Map` (or `Object.create(null)`), and assert a prototype key returns `undefined` in the test. (#129, #443)
- **When a runtime value becomes first-class, widen its TYPE — don't `as string` cast it.** Add the value to the canonical union so exhaustiveness checks stay live. (#257)
- **A per-value handler keyed off a `package.json` enum needs a gate over the MANIFEST enum, plus a neutral fallback.** A map/switch that happens to cover today's `speckit.aiProvider` values fails nothing when the enum gains an eleventh — TypeScript can't see the manifest. Assert every enum value resolves (and every asset it names exists on disk) in a test that reads `package.json`. And the `default:`/unmapped branch must land on a NEUTRAL value, never a specific sibling's identity — an unknown IDE-Chat host fell through to **Copilot's logo**, i.e. the extension shipped another vendor's brand for an unrecognized host. Same trap on the read side: renaming an enum value strands persisted old values, so readers need a legacy fallback. (#435, #257)
- **Type external API responses to the API's real schema** — mark genuinely-nullable fields `| null`, unconsumed/optional fields `?`; don't over-require `string`. (#274)
- **An inline config default MUST match the `package.json` manifest default.** A resolver fallback `['specs']` diverging from manifest `['specs', '.specify/specs']` reintroduces bugs on bare/early reads. Grep for the literal when you change a manifest default. (#277)
- **A settings migration rewrites ONLY known legacy values + coerces at read.** Migrate per-scope via `inspect()` + same-target `update()`; leave unknown strings for VS Code to flag; funnel every reader through one coercion helper (`Boolean("off") === true` is the trap). (#259)
- **A probe that answers "is this X?" must distinguish NO from I-COULDN'T-TELL — a bare `except`/`catch` returning the negative fails OPEN.** This shape recurred four times in one batch. `isProjectRoot()` returned `false` on any `statSync` error, so an unreadable nested config let the walk descend and re-report a sample project's specs as the parent's orphans; `os.path.isfile` swallowed `PermissionError` identically in the Python twin; and `_shallow_boundaries` returned an empty set on any read failure, meaning "not a shallow clone", which resurrected the exact false-success the change was fixing. Only the error that genuinely MEANS the negative may return it (`ENOENT`/`FileNotFoundError` = the file really is absent, which is git's/the config's own answer); every other error means unknown — return a third state (`None`) and have the caller skip loudly. **Scan for: `except OSError: return False/[]/frozenset()`, `catch { return false }`, and any `os.path.isfile`/`exists` used as a boundary or capability probe.** The tell is that the negative branch is also the "keep going" branch. (#455, #462, #464, #468)
- **When you RENAME a setting key, the readers must fall back to the legacy key — the migration alone is not enough.** The migration is best-effort (try/catch at activation); if it throws or hasn't run, the new key is unset (schema default) while the user's opt-in still lives on the old key, silently dropping it. Route every reader (incl. telemetry/snapshot reads) through one helper that prefers the new key when explicitly set, else coerces the legacy key(s). Grep the old key name — every live read must be the helper, not `config.get(newKey)`. (#307)

## Derived state & UI commands

- **One fact, one derivation — two paths to the same fact WILL disagree.** A tree row's icon and its tooltip computed from different sources drifted: the icon read the parent spec's lifecycle (completed → green check) while the tooltip read the document's own state ("In Progress"), so a `partial` document showed a check that contradicted its own hover. Same shape as a second module re-implementing an existing host-IDE detector (label vs. mark, identical today, free to drift tomorrow) and as a packing list restated in two docs (#432). **This includes re-deriving on teardown what you already knew at setup** — `closeInlineEditor()` re-found its target with `closest()`, but in row mode the container is a `<tbody>` inserted as a *sibling after* the row, so `closest()` never reached it and the row stayed `editing` forever; the host must remember the element it marked, not re-find it. **And it includes the anchor-vs-mount pair**: a drifted comment stored on line 2 but mounted on line 3 showed "line 2" in its header because the label read the persisted anchor instead of the line actually mounted onto — render from the resolved mount, keep the persisted anchor immutable (it is the *input* to re-anchoring, not its output). Flag any diff where two consumers each compute a fact instead of reading one resolved value: derive a single `DocumentState`/detector/manifest/handle and have both read it, so disagreement is unrepresentable rather than merely fixed. (#435, #432, #433)
- **An async command handler must RETURN its promise, not drop it.** `registerCommand('x', (p) => { provider.showAsync(p); })` swallows the rejection into an unhandled promise rejection and leaves VS Code unable to await the command — return the call instead of wrapping it in a block. When you find one, check its siblings: this shipped because the new handler copied the shape of a pre-existing one that had always been wrong, and "consistent with the broken neighbour" is not a defence — fix the whole family. (#434)
- **A command whose name asserts an end state must ENFORCE that state idempotently, never toggle.** `speckit.specs.collapseAll` and `expandAll` shared one toggle handler, so "Collapse All" on an already-collapsed tree *expanded* it — survivable as a single title-bar button, plainly wrong once both are named items in a quick-pick. Each named command sets its own end state unconditionally. While you're there, check the state SEED matches the real default (the same bug hid a stale `allCollapsed = false` under a comment claiming rows default to expanded — they default to collapsed). (#435)

## Webview rendering, dispatch & the comment layer

- **A raw-HTML passthrough must match a real emitted TAG shape, not a bare class substring.** The markdown renderer trusted any line that merely *contained* `class="living-` and emitted it unescaped, so an authored spec line with that substring in prose (`<img onerror=…> class="living-x"`) injected raw HTML/JS and could break feature-spec byte parity. `escapeHtml` does NOT protect a passthrough branch — it's not on that path. Anchor the check to the tag the preprocessor actually emits (`/^\s*<(?:div|span|ol|ul|li|p) class="living-/`), and prove authored prose containing the substring is escaped, not passed through. (#493)
- **A natural-language prompt dispatches via `executeInTerminal`, not `executeSlashCommand`.** `executeSlashCommand` forces a leading `/` on CLI-backed providers (`cliTerminalProvider`, gemini, wibey, claudeCode), so a prose instruction becomes `/<sentence…>` and fails as an unknown command — invisibly, because it "runs." Slash commands go through `executeSlashCommand`; free text goes through `executeInTerminal`. Flag any prose/AI-instruction string handed to `executeSlashCommand`. (#475)
- **Restore/mount an inline artifact by ELEMENT identity when its `data-line` can repeat.** `restoreComments` re-mounted each comment with a global `.line[data-line="N"]` query, but component lists (scenarios, uncovered files) reuse list-local `data-line` values, so a comment could mount on the first same-numbered line instead of the one its content resolved to. Have the re-anchor return the matched rendered line (carrying its DOM index) and mount that exact element; a bare number is not a unique handle. (#493)
- **Set global renderer flags at INIT before the first render, and on every navState update — not only on content messages.** A living spec's first paint rendered in feature-spec mode because `init` loaded `initialNav` into the signal but never called `setLivingMode`, and `navStateUpdated` never set it either — only `contentUpdated`/`viewerStateUpdated` did. Any surface that flips a module-level renderer flag must keep it in lockstep with navState everywhere the state can arrive. A Storybook story that sets such a flag must also reset it on unmount, or it leaks into the next story. (#493)

## Deleting / broadening a component

- **Deleting a component drops every capability it owned — confirm the replacement covers the SAME cases, not a subset.** Removing `GeneratingFooter` dropped manual "mark complete" for all steps, but the surviving backstop only covers implement. Enumerate what it did; verify each capability is re-homed. (#277)
- **Trace a deleted component's FULL message/handler/state chain.** Follow `postMessage` → handler → state fields → disk probes and remove/​re-home the whole chain, not just the `.tsx`. (#277)
- **A menu `when` that newly matches a contextValue needs a handler that ACTS on it.** Broadening a row's contextValue can surface a menu whose command no-ops → a dead, misleading click. Re-check every `view/item/context` `when` it now matches. (#257)
- **When you gate/change a UI list, gate EVERY assembler that feeds it — the source you edited may not be the one that renders.** The Create-Spec workflow picker is built independently in `workflowManager.getWorkflows()` AND `specEditorProvider.getWorkflows()`; gating only the former left the picker always showing Companion. Grep for the displayed entry/option across all builders; reuse one predicate (`isCompanionSelectable()`) rather than gating one copy. (#302)
- **A project-wide rename/scrub of a term misses in one direction and over-reaches in the other — check both.** *Missed:* `specs/<NNN>/spec.md` and `checklists/requirements.md` are copies of the command template made *before* you edited it, so run the scrub grep over `specs/<current>/` too, not just `src/`/`speckit-extension/`/`docs/` — **and over `CLAUDE.md` itself**, whose doc-map and examples outlive the thing they describe: the retired `templateProfile` setting was still named in three places, including the Changelog-voice section holding it up as the *exemplar* of "a setting users touch." A convention file that cites a deleted feature teaches every future agent the wrong thing, so grep it on every removal. (#311, #441) *Over-reached:* a blanket find/replace also rewrites files that use the old term for a **different thing** — renaming the "Spec Explorer" *view* retitled comments in `SpecExplorerProvider`, which is the *Specs* tree, so they then named the wrong tree. On any rename-heavy diff, read every hunk outside the renamed component and confirm the old term there referred to the thing being renamed. (#435)

## Shell & release scripts

- **Never interpolate a workspace path into a shell string** (`cd "${root}"`) — set the terminal's structured `cwd` via `createTerminal({ cwd })`. Use `echo`, not a `#` comment, for terminal hints (interactive zsh has `INTERACTIVE_COMMENTS` off). (#234)
- **`A && B || C` is NOT if-then-else** — C also runs when B fails. Use `if A; then B; else C; fi` for "do C only when probe A fails." (#273)
- **Single-quote shell strings containing backticks** (release notes/descriptions) — double quotes command-substitute them. (#273)
- **An invariant that must hold on every run needs idempotent re-assertion, not set-once-at-create** (e.g. `--prerelease` re-applied via `release edit` after an upsert). (#273)
- **A guarantee that holds only because the CALLER happens to do the right thing is not a guarantee — move it into the tool.** "The allow-list can't leak stale files" was true only because the publish flow `rm -rf`s its temp dir two steps earlier; any other caller (a re-run, a hand-invoked `--copy-to`) defeats it. When a tool's output has a stated property, the tool must enforce it — and enforce it *narrowly*: clear only what it owns (loose `*.py`, never recursive) and refuse a destination holding anything it doesn't recognize, rather than deleting broadly. (#432)
- **Swapping a convenience endpoint for a list endpoint drops what it did for free** — `/releases/latest` → `/releases` loses draft/prerelease filtering, pagination, and incidental headers (`User-Agent`); re-add each. (#274)

## Spec-context & lifecycle state

- **A script that takes a `--feature-dir`/`--dir` targeting a (possibly different) repo must derive git context — branch, repo root, changed files — from THAT path, not from cwd.** Deriving from `_repo_root()`/cwd silently records the wrong repo's branch when the target is a sandbox or sibling repo; use the path-anchored helper (`_repo_root_for(dir)`). (#363)
- **A writer that mutates a user's multi-block config file (e.g. `.specify/companion.yml`) must SPLICE only its own block — never re-emit the whole file from one block.** Re-rendering a single block as the entire file silently destroys sibling blocks (`commands:`/`hooks:`) and inter-block comments/blank lines. Replace just the target block's region, preserve everything else (incl. column-0 comments between blocks), and reject values the constrained reader can't escape (quotes/newlines) rather than emit corrupt YAML. (#365)
- **A write path that creates a missing artifact must create it well-formed, not a headerless fragment** (e.g. fold-back seeds a new living spec with a title + `## Requirements`, not an empty file) — and decide create-vs-skip deliberately against the feature's accumulation story. (#363)

- **Overriding a spec's `status` is not enough — realign the fields the UI actually derives from.** The viewer footer/step-tab key on `currentStep` + the derived `stepHistory`, NOT on `status` (`footerActions.ts`, `docs/viewer-states.md`). A writer that sets `status` to an in-progress phase but leaves `currentStep`/history on the old step yields an incoherent, still-stranded spec and a misleading "X completed" event. When forcing/overriding state, map the target to its owning step and record an honest `start`/`complete` boundary. (#347)
- **A shared lifecycle writer reused for a new case must not change byte-behavior for existing callers** — branch the new behavior (e.g. `forceStatus` for non-terminal overrides) and keep terminal `completed`/`archived` routing through the unchanged path; assert parity in a test. (#347)

## Paths & globs

- **A path constant in a registry (`PROVIDER_PATHS`, config defaults) must be PROVEN against what the emitting tool actually writes — a spec asserting it is not proof.** `PROVIDER_PATHS.codex.skillsDir` said `.codex/skills`; that directory has never existed (spec-kit's Codex integration emits skills to `.agents/skills/`), and it shipped because a research doc asserted the path and nobody looked. Two consequences to scan for: (a) the wrong constant becomes the *authority* — a reviewer (Copilot did) will "fix" correct code to match the lying registry, so when a review disputes a path, verify against the emitting source on disk, not against the constant; (b) any consumer that hardcodes a path instead of reading the registry drifts independently — `SkillManager.getSkillList()` hardcoded `.claude/skills`, so a Codex user's Skills sidebar listed **Claude's** skills and the watcher watched a nonexistent dir. Every path constant needs a test that the directory it names exists in the checked-in `specify init` fixtures; every consumer must read it from the registry. (#442)
- **A single `*` in a glob must not cross `/`.** Python `fnmatch` (and naive translations) let `*` match path separators, so `src/*.ts` wrongly claims `src/a/deep/x.ts`. Compile globs so `*`→`[^/]*` and `**`→`.*`, and make a trailing `/**` also match the bare directory. Normalize backslash inputs to `/` before matching. (#361)
- **A user-authored relative path joined onto a root for `fs`/`vscode.open` must be validated within that root.** A `spec:`/path from config or a discovered filename joined via `path.join(root, rel)` escapes the workspace when `rel` is absolute or contains `..` (`statSync`/open on `/etc/passwd` or `../secret.md`). Reject with a within-root check on `path.relative(root, path.resolve(root, rel))` — but reject only REAL traversal: `rel === '..'` or `rel.startsWith('..' + sep)` or `path.isAbsolute(rel)`. A naive `rel.startsWith('..')` also rejects a valid in-root name like `..config.yml`, and treating `rel === ''` (the root itself) as outside is wrong. DROP the offending entry rather than crash, and refuse to open it. (#380, #381)
- **A tree item's `collapsibleState` must reflect what `getChildren()` actually returns, not a proxy field — and derive it from cheap state, not by reading files on every refresh.** Keying "expandable?" on a sibling count (e.g. `tiers.length`) when the node always has an extra child makes a real-child node non-expandable. Conversely, doing synchronous YAML/disk reads in the node builder just to decide collapsibility blocks the UI on every refresh; when the node's children are fixed once a condition holds (installed → always has Config+Commands), gate collapsibility on that cheap flag and read lazily in `getChildren`. (#380, #381)
- **A `FileSystemWatcher` meant to track in-place updates must handle `onDidChange`, not just `onDidCreate`/`onDidDelete`.** A manifest/config rewritten in place (re-install, edit) fires only `onDidChange`; omitting it leaves the view stale until a create/delete. (#381)

## Tests & PR hygiene
- **Committed evidence / fixtures must not embed absolute or home paths or a username** (`/Users/<name>/…`) — record commands and paths **repo-relative**; it's noisy across machines and leaks local info. Regenerate captured artifacts from a real run, never hand-edit them to fake a result. (#361)
- **A test must exercise the REAL code path, not re-implement the condition under test.** A test that re-derives the predicate inline (e.g. recomputing a guard's boolean) passes even if the production guard is broken — false confidence. Extract the predicate into a named function and have both the production caller and the test call it. **The same trap on the INPUT side: feed the parser/resolver the string its real caller produces, not a hand-written ideal one.** The Codex command resolver was fixed and unit-tested against a bare `/speckit.companion.specify` — but the dispatch path that matters (`executeInTerminal`) prepends a context preamble, and the parser only read the first line, so the shipped fix was dead on every sidebar/viewer click while its tests were green. Ask of any parser test: *who calls this in production, and does the fixture match what THEY pass?* (#310, #442)
- **A `jest.mock` of a lifecycle/util module must export every symbol the module-under-test imports** — a missing export (e.g. `completeStep`) is `undefined` at call time and crashes only the path that reaches it, passing until that path is exercised. (#347)
- **A multi-path test must prove EACH path actually executed its branch, not just that the end state is right.** A path guarded by an idempotency check (`_is_more_advanced`, `CROSS_STEP_TERMINAL`, `_has_complete`) can silently no-op — return `None`/skip — so a test claiming to cover N writers may pass because ONE writer did all the work and the rest fell through the guard. Assert each call reached its branch (non-`None` return / a marker only that path writes), or order the paths so each is the first to act once. (#357)

- **A new gate/checker needs a test proving it FAILS — one per drift direction — not just that it passes when clean.** A suite where every case asserts the healthy state is green even if `check()` is hardcoded to return `[]`: the one behavior the gate exists for is untested. Feed it a synthetic broken input for each direction it claims to catch (missing entry / stale extra entry) and assert the specific failure. Same scan for the gate's own input derivation: if it **silently skips** an input it can't resolve (a `provides.commands[].file` absent from disk), it quietly shrinks the surface it scans — the exact drift class it exists to catch. Fail loud on an unresolvable input. (#432) **A *lock* test has the mirror-image failure: it must pin the property it claims to lock.** The manifest lock asserted only each menu item's group *prefix*, so items could silently reorder within a group — the exact drift the lock exists to prevent. Assert the full ordered list of ids, not membership or a prefix. (#440)
- **Green a stale test by re-deriving the fixture from the current contract — never weaken it to pass.** It must still fail on a real regression (verify by simulating one); append-only tests assert prior entries are byte-for-byte unchanged. Gate the suite in CI (no `|| true`). (#263)
- **When you broaden or narrow a change's scope mid-PR, update the spec/PR-description constraint in the SAME commit** — a diff that contradicts its own stated constraint is flagged immediately. This includes the spec artifacts: an `FR-`, `plan.md`, or `tasks.md` still asserting behavior the diff deliberately does NOT do (e.g. "bumps the version") is a defect in the diff, not a leftover. (#273, #432)
- **README/marketing images must not invent UI that misrepresents a concept** (e.g. showing a template profile as a model dropdown choice). (#234)
- **This repo has TWO packing lists, and a passing `package-manifest.py --check` only proves ONE of them.** `package-manifest.py` guards the **spec-kit archive**; `.vscodeignore` is the **VS Code extension's** separate list and is unguarded by it. `.vscodeignore` negates individual files by name (e.g. `write-context.py`), so **splitting a script into modules, or adding any runtime sibling it imports, silently drops the new files from the `.vsix`** — the packaged extension then dies with `ModuleNotFoundError` on first use while every gate stays green. Same class as #432, one list over. When a runtime script gains an import, update BOTH lists and confirm with `npx vsce ls`; prefer deriving each list from the entry point's real import closure over hand-maintaining names. (#432, #458, #470)
- **A release-critical LIST written as prose in a doc will drift — make it machine-readable and gate it.** Version literals, packing/allow-lists ("the archive ships these files"), command inventories: any of these hand-typed into a doc — *especially* duplicated across two docs (`publishing.md` + `publish-speckit-ext.md`) — is checked by nothing and the copy that actually runs is whichever one the operator opened. One source (`package-manifest.py`, `extension.yml`) consumed by both the builder and a CI gate; docs carry placeholders (`matches extension.yml` / `X.Y.Z`) and link to the source. Same scan for a repeated *narrative*: one canonical home for a postmortem/rationale, links from the rest. (#273, #432)
- **A feature branch must NOT bump `speckit-extension/extension.yml` `version` or date a CHANGELOG section.** `/publish-speckit-ext` owns the bump: the feature PR writes its entry under `[Unreleased]` and leaves `version`, the README badge, and `publishing.md` on the last released number. Flag any diff that bumps `extension.yml` while its CHANGELOG entry sits under `[Unreleased]` — the three copies then disagree and the README badge advertises a version no release exists for. (Same rule the root `package.json` already has for `/install-local`'s throwaway bump.) (#432)
